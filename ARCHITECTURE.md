# ICT Forex Bias — How Everything Works

A complete technical description of the project: the daily-bias data pipeline,
the Next.js frontend, the community features (auth + votes + chat), the live
economic calendar, theming, and how it all deploys.

---

## 1. What this is

A daily market-bias website built on **ICT / Smart Money** concepts. Every
trading day a bias is computed for 12 instruments (forex majors, indices,
commodities) and published as static JSON. The site renders that data with an
editorial "craft" design, plus live extras: a kill-zone session tracker, a
ForexFactory economic calendar, per-instrument **community voting**, and
per-instrument **discussion threads** behind sign-in.

- **Live site:** https://ict-forex-bias.vercel.app
- **Educational only — not financial advice.** (A risk banner is shown on every page.)

## 4. The data pipeline (how the bias is produced)

The bias is **not** computed in the browser or by the Next.js server. It is
generated by a Python pipeline and committed to the repo as JSON.

**Flow:** `generate_bias.py` orchestrates three steps:

1. **Fetch** — `fetch_data.fetch_all()` uses **yfinance** (Yahoo Finance) to pull
   daily, weekly, and hourly OHLC candles for all 12 instruments
   (EURUSD, GBPUSD, AUDUSD, NZDUSD, USDCAD, USDCHF, USDJPY, DXY, SPX500,
   XAUUSD, XAGUSD, USOIL — each mapped to its Yahoo ticker, e.g. `EURUSD=X`,
   `GC=F` for gold, `^GSPC` for S&P 500).

2. **Analyse** — `ict_analysis.analyse()` runs the **rule-based ICT engine**:
   market structure (BOS/CHoCH, swing highs/lows), premium/discount zones
   (equilibrium = 50% of the dealing range), order blocks, fair value gaps,
   liquidity pools (BSL/SSL, equal highs/lows), and a "draw on liquidity"
   target. It outputs weekly + daily bias, confidence (1–10), and key levels.
   The system is purely rule-based (chosen because it matched the backtested
   ~58.7% killzone win rate), so **no API key is required** to generate data.

3. **Write** — results are serialized to `data/bias/<YYYY-MM-DD>.json`, and
   `latest.json` + `index.json` are updated.

**Run it manually:**
```bash
cd scripts
python -m pip install -r requirements.txt
PYTHONUTF8=1 python generate_bias.py      # PYTHONUTF8=1 avoids a Windows console encoding crash on ▲/▼/◆
```

> ⚠️ On Windows the script prints bias arrows (▲ ▼ ◆); without `PYTHONUTF8=1`
> the cp1252 console throws `UnicodeEncodeError`. Linux/CI runners default to
> UTF-8 so the GitHub Action is unaffected.

## 5. Data model

`src/types/bias.ts` defines the shapes. Each daily file is a `DailyBias`:

```ts
DailyBias {
  date: string;              // "2026-06-16"
  generated_at: string;      // ISO timestamp
  market_overview: string;   // narrative paragraph
  instruments: InstrumentBias[];
}

InstrumentBias {
  symbol, current_price,
  weekly_bias, daily_bias,            // "Bullish" | "Bearish" | "Neutral"
  structure_label, last_bos,
  swing_high, swing_low, equilibrium,
  premium_discount,                   // "Premium" | "Discount" | "Equilibrium"
  previous_day_high/low, previous_week_high/low,
  nearest_ob, nearest_fvg,            // OrderBlock | FVG | null
  liquidity_levels: LiquidityLevel[], // {price, kind: BSL|SSL, equal}
  draw_on_liquidity, key_poi_label,
  confidence,                         // 1–10
  narrative
}
```

The frontend reads these JSON files directly from the filesystem at
build/request time (`readFileSync`), so the pages are static/ISR — no runtime DB
call is needed just to show the bias.

## 6. Frontend architecture (Next.js App Router)

Server vs client component split:

- **Server components** (default, can read files / fetch / use secrets):
  `page.tsx`, `archive/*`, `MarketOverview`, `StatsPill`, `BiasGrid`,
  `RiskWarning`, `NewsBlock`.
- **Client components** (`"use client"`, interactivity/state):
  `Header`, `KillzoneTracker`, `CategorySection`, `BiasCard`,
  `InstrumentVote`, `InstrumentChat`, `NewsBlockClient`, `AuthButtons`,
  and the providers (`ThemeContext`, `ChatContext`).

### Pages
- **`/` (home)** — reads `data/bias/latest.json`. Renders: risk warning →
  market overview → stats pill → kill-zone tracker → economic calendar →
  bias grid (the 12 instrument cards, grouped Forex/Index/Commodity).
- **`/archive`** — reads `index.json`, computes next-day outcomes (was the bias
  right?) and shows an accuracy table.
- **`/archive/[date]`** — reads one day's file (`generateStaticParams` +
  `dynamicParams = false`, so only known dates are built). Same layout as home
  but with per-instrument outcome badges. The live calendar is intentionally
  **not** shown here (a forward-looking "this week" calendar is meaningless on a
  historical date).

## 8. `src/lib` helpers

- **`flags.ts`** — `clerkEnabled = !!process.env.NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`.
  The whole community feature is gated on this so the site runs unchanged with
  no keys.
- **`ThemeContext.tsx`** — `dark`/`light` theme via a `data-theme` attribute on
  `<html>`, persisted to `localStorage`. A no-flash inline script in
  `layout.tsx` applies the stored theme before first paint.
- **`ChatContext.tsx`** — global chat-drawer state (`open(date, symbol)` /
  `close()`). Renders the floating chat window (§9) via a **React portal into
  `document.body`**.
- **`db.ts`** — Neon client (`DATABASE_URL` **or** `POSTGRES_URL`), plus
  `ensureSchema()` which idempotently creates the community tables on first use
  (so production never needs a manual migration).
- **`news.ts`** — `fetchEconomicCalendar()` (§10).
- **`translations.ts`** — UI strings (English only; the former Russian locale was
  removed).

## 9. Overlays (theming-safe floating windows)

Two overlays render via **`createPortal(..., document.body)`** so no
transformed/filtered ancestor can break their `position: fixed`:

- **Full-analysis modal** (`BiasCard`) — clicking a card opens a **centered**
  overlay (flex-centered, not transform-centered, so the entry animation can't
  fight the centering) with all detail sections: community vote, "Open
  conversation" button, key levels, market structure, POI, liquidity, narrative.
  The card grid stays at neutral height — the analysis floats above it, so
  expanding one card never pushes the others. Background scroll is locked and
  `scrollbar-gutter: stable` prevents a layout shift.
- **Chat drawer** (`ChatContext`) — "Open conversation" opens a floating window
  fixed to the **right** side (400px on desktop, full-width on mobile) holding
  `InstrumentChat`. Closes via ✕ / backdrop / Escape.

> Gotcha fixed here: Tailwind only generates classes it finds in the files
> listed in `tailwind.config.ts` `content`. That glob now covers **all of
> `src/**`** — originally `src/lib` was excluded, so the drawer's classes
> (`fixed`, `right-4`, …) weren't generated and it rendered unstyled at the
> bottom of the page.

## 10. Economic calendar (ForexFactory)

`src/lib/news.ts` fetches the **FairEconomy JSON feed** that powers
ForexFactory's calendar:

```
https://nfs.faireconomy.media/ff_calendar_thisweek.json
```

- Cached 30 min via Next's fetch cache (`next: { revalidate: 1800 }`).
- Maps `{title, country, date, impact, forecast, previous}` → events; keeps
  High/Medium/Low (drops Holiday); stores each time as an epoch `ts`.
- `NewsBlockClient` converts `ts` to the user's selected timezone, **starts the
  list at today** (filters `dayNum(ts) >= today` in the chosen tz, gated behind
  mount to avoid hydration mismatch), groups by day, and offers impact +
  currency filters behind the gear icon. Filter/timezone choices persist to
  `localStorage`.
- No API key; if the feed is unreachable the block degrades to an empty state.

## 11. Community features (auth + votes + chat)

All gated by `clerkEnabled`. With no keys, the widgets show a "activates once
configured" placeholder and the rest of the site is unaffected.

- **Auth:** Clerk. `layout.tsx` wraps the app in `<ClerkProvider>` only when
  configured; `middleware.ts` runs `clerkMiddleware()` only when configured
  (otherwise a no-op). Routes stay public — sign-in is enforced per **action**,
  not per route.
- **Storage:** Postgres via server actions in `src/app/actions/community.ts`.
  Each action checks `ready()` (DB + Clerk configured), calls `ensureSchema()`,
  reads the Clerk `userId` from `auth()`, then queries.
- **Tables** (`schema.sql`, also self-created by `ensureSchema`):
  - `instrument_votes(date, symbol, user_id, direction, …)` — PK `(date, symbol, user_id)`, so **one vote per user per instrument per day** (upsert to change).
  - `instrument_chat(id, date, symbol, user_id, author, body, created_at)` — threaded messages per instrument per day.
- **Votes** → `InstrumentVote` (sentiment bar + buttons) inside the analysis overlay.
- **Chat** → "Open conversation" → right-side drawer → `InstrumentChat`
  (loads + posts via server actions, polls every 5s).

Setup steps are in **`COMMUNITY_SETUP.md`**.

## 12. Theming

- `globals.css` defines the **craft palette** as space-separated RGB channels in
  CSS variables (`--bg`, `--accent`, `--bullish`, …), with a light override
  under `:root[data-theme="light"]`.
- `tailwind.config.ts` maps every color token to `rgb(var(--x) / <alpha-value>)`,
  so Tailwind opacity modifiers (`bg-card/70`) work **and** a single
  `data-theme` swap re-themes the entire app.
- Headings use the **Fraunces** serif (`.display` / `.serif`); data/labels use
  IBM Plex Mono. Root font-size is `108%` to scale the whole UI up slightly.

## 15. Deployment & the daily automation

- **Hosting:** Vercel. Deploys are currently done with `vercel --prod` from the
  local folder, which uploads the local snapshot (including `data/bias`).
- **Env vars:** Clerk keys + the Neon `DATABASE_URL` are set in the Vercel
  project (Neon's Vercel integration injects the DB vars automatically).
- **Daily automation:** `.github/workflows/daily-bias.yml` — cron
  `0 6 * * 1-5` (06:00 UTC, weekdays). It checks out the repo, runs
  `generate_bias.py`, commits new `data/bias/*.json`, and pushes. Vercel can
  auto-deploy on that push if the project is connected to GitHub.

### ⚠️ Known maintenance notes
1. **Daily bias depends on the Action committing + that commit reaching
   production.** If you deploy via `vercel --prod` from a stale local clone, you
   ship old data regardless of what the Action did. The robust fix is to connect
   Vercel to the GitHub repo so the Action's daily commit auto-deploys.
2. The bias data had a gap (Apr 28 – Jun 15) because the generator only produces
   "today"; the archive jumps from Apr 27 to Jun 16. Backfill by running the
   generator per missing date if desired.
3. GitHub disables cron workflows after ~60 days of repo inactivity — keep the
   repo active or re-enable in the Actions tab.
4. **Security:** the git `origin` URL had an embedded GitHub token; it should be
   revoked and the remote reset to a credential-free URL.

> Python computes the daily bias from Yahoo price data → writes JSON into the
> repo → Next.js renders that JSON with a craft-themed UI → Clerk + Neon add
> per-instrument community voting and chat on top → ForexFactory's feed powers a
> live economic calendar → everything ships to Vercel.
